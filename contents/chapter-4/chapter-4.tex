\chapter{Perancangan dan Implementasi}

\section{Ringkasan Sistem}

Bab ini menjelaskan implementasi inti dari peripheral \textit{I2S transmitter} yang
dirancang sebagai custom IP AXI4-Lite. Fokus rancangan adalah mengubah sampel PCM
stereo yang ditulis oleh MicroBlaze V melalui register menjadi sinyal serial
\path{i2s_mclk}, \path{i2s_bclk}, \path{i2s_ws}, dan \path{i2s_data} sesuai format Philips
I2S~\citep{philips_i2s_1986}. Implementasi utama berada pada tiga bagian:
\path{ip_repo/i2s_1_0/hdl} untuk RTL IP, kode aplikasi C melalui Vitis, dan \texttt{testblock}
sebagai testbench RTL.

        \includegraphics[width=0.92\textwidth]{contents/figures/i2s-ip-block.png}%
    }{}
    \caption{Blok internal IP I2S TX berbasis AXI4-Lite.}
    \label{fig:ch4-block-ip}

Alur data dimulai dari aplikasi C yang menulis tiga register aktif, yaitu
\path{DATA_LEFT}, \path{DATA_RIGHT}, dan \path{CONTROL}. Register tersebut berada pada domain
\path{S_AXI_ACLK} 100 MHz, kemudian disinkronkan ke domain clock audio. Serializer
I2S membaca sampel pada awal frame stereo agar data tidak berubah di tengah word,
lalu mengirimkan kanal kiri dan kanan masing-masing dalam slot 32 bit.

\section{Spesifikasi IP I2S}

\subsection{Fitur utama}
\begin{itemize}
    \item Antarmuka kontrol \textbf{AXI4-Lite 32 bit} untuk akses dari MicroBlaze V~\citep{arm_amba_axi_2010}.
    \item Mode \textbf{TX-only stereo} dengan format \textbf{Philips I2S}, MSB-first, satu bit delay setelah perubahan \texttt{WS}.
    \item Lebar sampel dapat dipilih 16, 24, atau 32 bit melalui register \texttt{CONTROL}.
    \item Dua keluarga clock audio: 48/96 kHz dan 44,1/88,2 kHz.
    \item Tidak menggunakan FIFO; firmware menulis sampel secara \textit{programmed I/O}.
\end{itemize}

\section{Implementasi RTL IP}

RTL IP terdiri dari \path{i2s.v} sebagai wrapper dan
\path{i2s_slave_lite_v1_0_S00_AXI.v} sebagai modul utama. Wrapper hanya
menjaga nama port agar sesuai dengan IP Packager Vivado, sedangkan modul utama
berisi register AXI4-Lite, pemilihan clock, CDC, pembagi clock, pengambilan sampel,
dan serializer I2S.

\subsection{Wrapper IP}

\begin{lstlisting}[language=Verilog, caption={Wrapper \texttt{i2s.v} meneruskan port AXI dan I2S ke modul utama.}, label={lst:ch4-wrapper}]
i2s_slave_lite_v1_0_S00_AXI #(
    .C_S_AXI_DATA_WIDTH(C_S_AXI_DATA_WIDTH),
    .C_S_AXI_ADDR_WIDTH(C_S_AXI_ADDR_WIDTH)
) i2s_slave_lite_v1_0_S00_AXI_inst (
    .audio_48_clk(audio_48_clk),
    .audio_44_clk(audio_44_clk),
    .i2s_mclk(i2s_mclk),
    .i2s_bclk(i2s_bclk),
    .i2s_ws(i2s_ws),
    .i2s_data(i2s_data),
    .S_AXI_ACLK(S_AXI_ACLK),
    .S_AXI_ARESETN(S_AXI_ARESETN)
);
\end{lstlisting}

Cuplikan \ref{lst:ch4-wrapper} menunjukkan bahwa wrapper tidak menambah logika
baru. Tujuannya adalah memisahkan antarmuka IP yang dilihat Vivado dari logika
internal, sehingga perubahan pada serializer tidak mengubah struktur integrasi
di Block Design.

\subsection{Register AXI4-Lite}

\begin{lstlisting}[language=Verilog, caption={Register file AXI4-Lite untuk data dan kontrol.}, label={lst:ch4-regfile}]
reg [C_S_AXI_DATA_WIDTH-1:0] slv_reg0;  // DATA_LEFT
reg [C_S_AXI_DATA_WIDTH-1:0] slv_reg1;  // DATA_RIGHT
reg [C_S_AXI_DATA_WIDTH-1:0] slv_reg2;  // CONTROL

localparam [1:0] REG_DATA_LEFT  = 2'h0;
localparam [1:0] REG_DATA_RIGHT = 2'h1;
localparam [1:0] REG_CONTROL    = 2'h2;
\end{lstlisting}

Register \texttt{slv\_reg0} dan \texttt{slv\_reg1} menyimpan sampel kiri dan kanan.
\texttt{slv\_reg2} menyimpan konfigurasi operasi. Dengan hanya tiga register
aktif, firmware cukup melakukan akses MMIO sederhana tanpa protokol streaming
tambahan maupun antarmuka pemantauan khusus.

\subsection{Pemilihan clock audio}

\begin{lstlisting}[language=Verilog, caption={Pemilihan keluarga clock audio.}, label={lst:ch4-clkmux}]
wire reg_fs_family = slv_reg2[2];
wire internal_mclk_unbuf;
wire internal_mclk;

clk_mux mclk_mux_inst (
    .clk0  (audio_48_clk),
    .clk1  (audio_44_clk),
    .sel   (reg_fs_family),
    .clk_o (internal_mclk_unbuf)
);
\end{lstlisting}

Bit \texttt{FS\_FAMILY} memilih clock 24,576 MHz untuk keluarga 48/96 kHz atau
22,579 MHz untuk keluarga 44,1/88,2 kHz. Modul \texttt{clk\_mux} memakai enable
yang disinkronkan pada sisi negatif clock, sehingga perpindahan clock tidak
menghasilkan pulsa pendek yang dapat merusak logika serializer.

\subsection{Clock domain crossing}

\begin{lstlisting}[language=Verilog, caption={Sinkronisasi register dari domain AXI ke domain audio.}, label={lst:ch4-cdc}]
cdc_sync #(.WIDTH(32)) cdc_ctrl_inst (
    .dest_clk (internal_mclk),
    .src_in   (slv_reg2),
    .dest_out (ctrl_cdc_w)
);

cdc_sync #(.WIDTH(32)) cdc_left_inst (
    .dest_clk (internal_mclk),
    .src_in   (slv_reg0),
    .dest_out (sample_left_cdc_w)
);
\end{lstlisting}

Register AXI ditulis pada \path{S_AXI_ACLK}, sedangkan serializer berjalan pada
\path{internal_mclk}. Karena kedua domain tidak identik, setiap word register
disinkronkan dengan dua tahap flip-flop. Sampel kemudian baru diambil pada
batas frame stereo, sehingga risiko perubahan data di tengah pengiriman kanal
dapat dihindari.

\subsection{Pembagi clock dan pengambilan sampel}

\begin{lstlisting}[language=Verilog, caption={Pembagi BCLK dan pengambilan sampel pada awal frame.}, label={lst:ch4-divider-sample}]
wire bclk_en_w = fs_mode_sync ? (clock_div_q[0] == 1'b0)
                              : (clock_div_q[1:0] == 2'd0);

always @(posedge internal_mclk or posedge audio_rst) begin
    if (audio_rst) begin
        sample_left_q  <= 32'd0;
        sample_right_q <= 32'd0;
    end else if (bclk_en_w && !bclk_q && (bit_count_q == 6'd0)) begin
        sample_left_q  <= sample_left_cdc_w;
        sample_right_q <= sample_right_cdc_w;
    end
end
\end{lstlisting}

\texttt{bclk\_en\_w} menentukan kapan state I2S bergerak. Saat
\texttt{FS\_MODE=0}, BCLK efektif lebih rendah; saat \texttt{FS\_MODE=1}, BCLK
dipercepat untuk menghasilkan Fs dua kali lipat. Pengambilan sampel dilakukan ketika
\texttt{bit\_count\_q=0}, yaitu awal frame stereo, agar satu frame selalu memakai
pasangan sampel kiri-kanan yang konsisten.

\subsection{Serializer Philips I2S}

\begin{lstlisting}[language=Verilog, caption={Fungsi pemetaan sampel paralel menjadi bit serial I2S.}, label={lst:ch4-serializer}]
function i2s_next_serial_bit;
    input [31:0] left_samp;
    input [31:0] right_samp;
    input [5:0]  bc;
    input [5:0]  width;
    begin
        in_left = (bc < 6'd32);
        pos     = bc[4:0];
        if (width == 6'd0)
            i2s_next_serial_bit = 1'b0;
        else if (pos == 5'd0)
            i2s_next_serial_bit = 1'b0;
        else if (pos <= width) begin
            bit_idx = width - {1'b0, pos};
            i2s_next_serial_bit = in_left ? left_samp[bit_idx[4:0]]
                                          : right_samp[bit_idx[4:0]];
        end else
            i2s_next_serial_bit = 1'b0;
    end
endfunction
\end{lstlisting}

Nilai \texttt{bc} 0--31 digunakan untuk kanal kiri dan 32--63 untuk kanal kanan.
\texttt{pos==0} selalu mengeluarkan nol sebagai delay bit Philips I2S. Setelah
itu, data dikirim MSB-first sesuai \texttt{SAMPLE\_WIDTH}; bit di luar lebar
sampel diisi nol sebagai padding slot 32 bit.

\subsection{Generator output I2S}

\begin{lstlisting}[language=Verilog, caption={Generator BCLK, WS, dan DATA.}, label={lst:ch4-output-gen}]
always @(posedge internal_mclk or posedge audio_rst) begin
    if (audio_rst) begin
        bit_count_q <= 6'd0;
        data_q <= 1'b0; ws_q <= 1'b0; bclk_q <= 1'b0;
    end else if (bclk_en_w) begin
        if (!enable_sync) begin
            bit_count_q <= 6'd0;
            data_q <= 1'b0; ws_q <= 1'b0; bclk_q <= 1'b0;
        end else if (bclk_q) begin
            bclk_q <= 1'b0;
            data_q <= i2s_next_serial_bit(sample_for_i2s_left,
                                          sample_for_i2s_right,
                                          bit_count_q,
                                          sample_width_sync);
            ws_q <= bit_count_q[5];
            bit_count_q <= bit_count_q + 6'd1;
        end else begin
            bclk_q <= 1'b1;
        end
    end
end
\end{lstlisting}

Data dan \texttt{WS} diperbarui ketika \texttt{BCLK} turun, sedangkan DAC membaca
data pada sisi naik \texttt{BCLK}. Dengan demikian terdapat waktu setup yang
cukup antara perubahan data dan pengambilan data oleh DAC. Jika
\texttt{ENABLE=0}, semua output ditahan rendah. Jika \texttt{MUTE=1}, clock tetap
berjalan tetapi sampel yang masuk ke serializer dipaksa nol.

\section{Implementasi Software Vitis pada MicroBlaze}

Aplikasi \texttt{helloworld.c} berperan sebagai program uji bare-metal. Program
ini menguji koneksi AXI, mengatur register \texttt{CONTROL}, menghasilkan sampel
sinus, dan mengirim sampel ke register data dengan interval tetap.

\subsection{Definisi alamat register}

\begin{lstlisting}[language=C, caption={Alamat register IP pada firmware.}, label={lst:ch4-fw-reg}]
#define I2S_BASE       XPAR_I2S_0_BASEADDR
#define DATA_LEFT      (I2S_BASE + 0x00U)
#define DATA_RIGHT     (I2S_BASE + 0x04U)
#define CONTROL        (I2S_BASE + 0x08U)

#define CTRL_ENABLE    (1U << 0)
#define CTRL_MUTE      (1U << 1)
#define CTRL_FS        (1U << 2)
#define CTRL_FS_MODE   (1U << 3)
#define CTRL_WIDTH(w)  (((uint32_t)(w) & 0x1FU) << 8U)
\end{lstlisting}

Firmware tidak mengakses sinyal RTL secara langsung. Semua kontrol dilakukan
dengan \texttt{Xil\_Out32()} dan \texttt{Xil\_In32()} pada alamat register yang
dibuat oleh Vivado melalui \texttt{xparameters.h}.

\subsection{Pembentukan register kontrol}

\begin{lstlisting}[language=C, caption={Penggabungan state software menjadi word \texttt{CONTROL}.}, label={lst:ch4-fw-control}]
static void apply_control(void)
{
    uint32_t ctrl = 0U;
    if (g_enable)  ctrl |= CTRL_ENABLE;
    if (g_mute)    ctrl |= CTRL_MUTE;
    if (g_fs)      ctrl |= CTRL_FS;
    if (g_fs_mode) ctrl |= CTRL_FS_MODE;
    ctrl |= CTRL_WIDTH(g_width);
    Xil_Out32(CONTROL, ctrl);
}
\end{lstlisting}

Fungsi ini menjadi satu titik konfigurasi IP. Ketika pengguna menekan tombol
UART untuk mengubah \texttt{ENABLE}, \texttt{MUTE}, keluarga Fs, mode Fs, atau
lebar sampel, firmware memperbarui variabel state lalu menulis ulang
\texttt{CONTROL}.

\subsection{Pembangkitan dan penulisan sampel}

\begin{lstlisting}[language=C, caption={Pengiriman sampel sinus ke register data.}, label={lst:ch4-fw-sample}]
static void stream_sample(void)
{
    static uint32_t phase = 0U;
    uint32_t sr   = current_sample_rate();
    uint32_t step = (uint32_t)(((uint64_t)g_freq << 32) / sr);

    int16_t s = (int16_t)(((int32_t)SINE[phase >> 27] * AMPLITUDE) / 32767);
    int32_t s32 = (g_width >= 32U) ? ((int32_t)s << 16) :
                  (g_width > 16U)  ? ((int32_t)s << (g_width - 16U)) :
                  (g_width < 16U)  ? ((int32_t)s >> (16U - g_width)) :
                                      (int32_t)s;

    Xil_Out32(DATA_LEFT,  g_left  ? (uint32_t)s32 : 0U);
    Xil_Out32(DATA_RIGHT, g_right ? (uint32_t)s32 : 0U);
    phase += step;
}
\end{lstlisting}

\texttt{phase} adalah akumulator fase 32 bit. Nilai \texttt{step} menentukan
frekuensi nada yang dihasilkan. Sampel sinus 16 bit digeser agar MSB berada
pada posisi \texttt{SAMPLE\_WIDTH-1}, sesuai asumsi serializer RTL. Kanal yang
dinonaktifkan ditulis nol, tetapi kedua register tetap ditulis setiap periode
agar pasangan kanal tetap konsisten.

\subsection{Loop timing berbasis \texttt{mcycle}}

\begin{lstlisting}[language=C, caption={Loop sampel berbasis deadline absolut.}, label={lst:ch4-fw-deadline}]
uint64_t t_deadline = rdcycle64();

while (1) {
    uint32_t sample_rate   = current_sample_rate();
    uint64_t period_cycles =
        (uint64_t)XPAR_CPU_CORE_CLOCK_FREQ_HZ / sample_rate;

    stream_sample();
    t_deadline += period_cycles;
    while (rdcycle64() < t_deadline) {}
}
\end{lstlisting}

Loop ini menggunakan counter \texttt{mcycle} MicroBlaze V sehingga periode sampel
ditentukan oleh jumlah siklus CPU, bukan oleh delay berbasis fungsi tidur.
Pendekatan deadline absolut mengurangi drift karena setiap periode dihitung
dari jadwal ideal, bukan dari waktu selesai iterasi sebelumnya.

\section{Testbench RTL}

Direktori \texttt{testblock} berisi versi testbench yang menyederhanakan akses
AXI menjadi sinyal langsung: \texttt{enable}, \texttt{mute}, \texttt{fs\_family},
\texttt{fs\_mode}, \texttt{sample\_width}, \texttt{sample\_l}, dan \texttt{sample\_r}.
Tujuannya adalah memverifikasi serializer I2S tanpa perlu mensimulasikan seluruh
transaksi AXI.

\subsection{Model IP pada testbench}

\begin{lstlisting}[language=Verilog, caption={Model \texttt{i2s\_ip.v} pada proyek testblock.}, label={lst:ch4-tb-ip}]

    (fs_family == 1'b0) ? audio_48_clk
                        : audio_44_clk;

    (fs_mode == 1'b1)
        ? (clock_div_q[0] == 1'b0)
        : (clock_div_q[1:0] == 2'b00);
\end{lstlisting}

Model ini mempertahankan perilaku penting IP final, yaitu pemilihan keluarga
clock dan perubahan pembagi BCLK berdasarkan \texttt{fs\_mode}. Perbedaannya,
input kontrol diberikan langsung oleh testbench, bukan melalui AXI4-Lite.

\subsection{Skenario stimulus}

\begin{lstlisting}[language=Verilog, caption={Stimulus utama pada \texttt{i2s\_tb.v}.}, label={lst:ch4-tb-stimulus}]
initial begin
    sample_l = 32'h0000AAAA;
    sample_r = 32'h00005555;
    enable = 0;
    mute = 0;
    fs_mode = 0;

    #2000; enable = 1;
    #5000; mute = 1;
    #5000; mute = 0; fs_mode = 1;
    #1000000; $finish;
end
\end{lstlisting}

Nilai \texttt{AAAA} dan \texttt{5555} dipilih karena pola bitnya mudah diamati
pada waveform. Skenario dimulai dari kondisi nonaktif, kemudian menguji output
normal, kondisi mute, dan perubahan mode Fs. Simulasi dinyatakan sesuai apabila
\texttt{WS} membagi frame menjadi kanal kiri-kanan, \texttt{DATA} memiliki satu
bit delay setelah perubahan \texttt{WS}, dan \texttt{BCLK} berubah mengikuti
\texttt{fs\_mode}.

\section{Ringkasan Implementasi}

Implementasi akhir membentuk rantai lengkap dari software ke hardware. Firmware
menulis sampel dan kontrol melalui AXI4-Lite, RTL menyinkronkan register ke
domain audio, pengambilan sampel dilakukan pada awal frame, dan serializer mengirim
data dalam format Philips I2S. Testbench RTL memvalidasi perilaku inti serializer,
sedangkan pengujian hardware pada Bab V memverifikasi bahwa sinyal tersebut
dapat diterima oleh DAC PCM5102A.

\section{Optimasi Desain: IP I2S Vendor-Agnostic dengan Clock Synthesis DDS}

Sebagai langkah pengembangan lebih lanjut, dilakukan optimasi pada desain IP I2S untuk mengatasi ketergantungan pada sumber clock audio eksternal dan ketergantungan pada primitif perangkat keras tertentu (\textit{vendor-specific}). Desain baru ini menggunakan teknik \textit{Direct Digital Synthesis} (DDS) pada sisi perangkat keras yang dikhususkan untuk pembangkitan clock audio secara internal langsung dari domain clock sistem AXI (100 MHz).

Penting untuk membedakan dua implementasi DDS dalam sistem ini:
\begin{itemize}
    \item \textbf{DDS pada RTL}: Digunakan untuk sintesis frekuensi clock (\texttt{BCLK}, \texttt{MCLK}, \texttt{WS}) agar mendukung frekuensi sampling (\textit{sampling rate}) arbitrer.
    \item \textbf{DDS pada Software}: Digunakan untuk pembangkitan sinyal audio (misalnya gelombang sinus) yang menghasilkan sampel PCM untuk ditulis ke register hardware.
\end{itemize}

\subsection{Pembangkitan Clock Berbasis DDS pada RTL}

Berbeda dengan desain awal yang memerlukan input \texttt{audio\_48\_clk} dan \texttt{audio\_44\_clk} dari luar IP, varian teroptimasi ini melakukan sintesis frekuensi secara internal menggunakan akumulator fase 32-bit. Teknik ini memungkinkan pembangkitan \texttt{BCLK} dan \texttt{MCLK} dengan resolusi yang sangat tinggi tanpa memerlukan blok PLL atau MMCM tambahan di luar IP.

Persamaan dasar yang digunakan untuk menghitung nilai penambahan fase (\textit{frequency word}) adalah sebagai berikut:

\begin{equation}
    \Delta \text{phase} = \frac{f_{\text{target}}}{f_{\text{system}}} \times 2^{32}
    \label{eq:dds-formula}
\end{equation}

Dalam implementasi RTL, pembagian clock dilakukan dengan mendeteksi *overflow* pada akumulator fase terhadap frekuensi sistem AXI. Hal ini membuat desain bersifat \textit{vendor-agnostic} karena tidak lagi menggunakan primitif Xilinx seperti \texttt{BUFGMUX} atau \texttt{BUFG}, sehingga kode Verilog dapat langsung diimplementasikan pada FPGA dari manufaktur lain tanpa perubahan.

\subsection{Implementasi RTL Varian DDS}

Bagian ini membahas detail implementasi pada berkas \texttt{i2s\_dds\_slave\_lite\_v1\_0\_S00\_AXI.v}. Perubahan mendasar terletak pada cara logika serializer digerakkan oleh akumulator DDS dan penambahan mekanisme *handshake* melalui register status.

\subsubsection{Sintesis Frekuensi Clock Audio}

Logika DDS bekerja dengan menambahkan nilai inkremen (\texttt{toggle\_rate}) ke akumulator setiap siklus clock AXI. Ketika akumulator melewati nilai frekuensi sistem, sinyal clock output dibalik (\textit{toggle}).

\begin{lstlisting}[language=Verilog, caption={Logika perhitungan inkremen DDS untuk BCLK dan MCLK.}, label={lst:ch4-dds-inc}]
// bclk_toggle_rate = Fs x 64 x 2 (untuk toggle) = Fs x 128
wire [31:0] bclk_toggle_rate_hz = {sample_rate_sync, 7'b0};
// mclk_toggle_rate = Fs x 256 x 2 (untuk toggle) = Fs x 512
wire [31:0] mclk_toggle_rate_hz = {sample_rate_sync, 9'b0};

wire [31:0] bclk_phase_sum = bclk_phase_acc_q + bclk_toggle_rate_hz;
wire bclk_phase_wrap = (bclk_phase_sum >= AXI_CLK_HZ);
\end{lstlisting}

Penggunaan operator geser (\texttt{<< 7} dan \texttt{<< 9}) memungkinkan perhitungan rasio clock dilakukan secara efisien dalam hardware tanpa menggunakan pembagi (\textit{divider}) yang berat. Nilai \texttt{bclk\_phase\_wrap} menjadi sinyal kontrol yang memberitahu serializer kapan harus mengubah status bit data atau clock.

\subsubsection{Serializer yang Digerakkan DDS}

Berbeda dengan desain awal yang menggunakan pembagi integer tetap, desain ini menggunakan \texttt{bclk\_phase\_wrap} sebagai \textit{clock enable} di dalam domain \texttt{S\_AXI\_ACLK}.

\begin{lstlisting}[language=Verilog, caption={Sekuenser I2S berbasis DDS.}, label={lst:ch4-dds-seq}]
always @(posedge S_AXI_ACLK) begin
    if (!S_AXI_ARESETN) begin
        // Reset state
    end else if (enable_sync) begin
        if (bclk_phase_wrap) begin
            bclk_phase_acc_q <= bclk_phase_sum - AXI_CLK_HZ;
            if (bclk_q) begin
                bclk_q      <= 1'b0; // Falling edge
                bit_count_q <= bit_count_q + 6'd1;
                ws_q        <= bit_count_q[5];
                data_q      <= i2s_next_serial_bit(...);
            end else begin
                bclk_q <= 1'b1; // Rising edge
                if (bit_count_q == 6'd0) begin
                    sample_left_q  <= slv_reg0;
                    sample_right_q <= slv_reg1;
                    current_latch_status_q <= !current_latch_status_q;
                end
            end
        end
    end
end
\end{lstlisting}

Pada Cuplikan \ref{lst:ch4-dds-seq}, terlihat bahwa pengambilan sampel (\textit{latching}) dilakukan tepat saat \texttt{bit\_count\_q == 0} pada sisi naik BCLK pertama dalam satu frame. Hal ini menjamin sinkronisasi yang sempurna antara data yang ditulis perangkat lunak dan data yang dikirimkan secara serial.

\subsubsection{Mekanisme Feedback Status}

Untuk memudahkan sinkronisasi pada sisi perangkat lunak, ditambahkan bit status yang bisa dibaca melalui AXI.

\begin{lstlisting}[language=Verilog, caption={Pemetaan register status untuk feedback ke software.}, label={lst:ch4-dds-status}]
case (read_addr)
    REG_DATA_LEFT:  reg_data_out = slv_reg0;
    REG_DATA_RIGHT: reg_data_out = slv_reg1;
    REG_CONTROL:    reg_data_out = slv_reg2 & CONTROL_MASK;
    REG_STATUS:     reg_data_out = {slv_reg3[31:1], current_latch_status_q};
    default:        reg_data_out = 32'd0;
endcase
\end{lstlisting}

Bit \texttt{current\_latch\_status\_q} akan berganti nilai setiap kali IP mengambil pasangan sampel baru dari register data. Perangkat lunak dapat memantau perubahan bit ini untuk memastikan bahwa penulisan sampel berikutnya dilakukan pada waktu yang tepat. Penggunaan register status ini sangat krusial sebagai mekanisme \textit{handshake} untuk mencegah terjadinya \textit{buffer underrun} (suara terputus) atau \textit{buffer overrun} (data tertimpa) karena ketidaksesuaian laju antara eksekusi prosesor dan kecepatan transmisi perangkat keras.

\section{Implementasi Perangkat Lunak Teroptimasi}

Dengan adanya fitur DDS dan register status pada sisi perangkat keras, implementasi perangkat lunak pada Vitis (\texttt{helloworld.c}) juga mengalami perubahan signifikan. Fokus utama perangkat lunak beralih dari sekadar mengatur waktu tunggu statis menjadi sistem yang sinkron terhadap status perangkat keras.

\subsection{Sinkronisasi Berbasis Status Latch}

Berbeda dengan implementasi awal yang menggunakan loop berbasis \textit{deadline} siklus CPU, perangkat lunak terbaru menggunakan metode \textit{polling} pada register status untuk mendeteksi kapan perangkat keras siap menerima sampel baru.

\begin{lstlisting}[language=C, caption={Loop sinkronisasi perangkat lunak terhadap status frame hardware.}, label={lst:ch4-sw-sync}]
while (1) {
    uint32_t frame_bit = Xil_In32(STATUS_REG) & 1U;

    // Deteksi perubahan bit status (toggle)
    if (g_enable && frame_bit != g_last_frame_bit) {
        g_last_frame_bit = frame_bit;
        stream_software_dds_sample(); // Kirim sampel berikutnya
    }

    // Cek input UART tanpa memblokir audio
    if (!XUartLite_IsReceiveEmpty(STDIN_BASEADDRESS))
        handle_key((uint8_t)XUartLite_RecvByte(STDIN_BASEADDRESS));
}
\end{lstlisting}

Metode ini menjamin bahwa setiap sampel yang dihasilkan oleh perangkat lunak dikirimkan tepat pada saat transisi frame I2S dimulai, sehingga menghasilkan fidelitas audio yang jauh lebih stabil meskipun frekuensi sampling diubah secara dinamis.

\subsection{Pembangkitan Sinyal DDS pada Software}

Pada varian ini, pembangkitan sinyal audio dilakukan sepenuhnya dalam perangkat lunak menggunakan akumulator fase. Hal ini memberikan fleksibilitas bagi peneliti untuk menghasilkan berbagai jenis gelombang (sinus, kotak, dsb.) tanpa harus mengubah desain RTL.

\begin{lstlisting}[language=C, caption={Pembangkitan gelombang audio menggunakan DDS pada software.}, label={lst:ch4-sw-dds}]
static void stream_software_dds_sample(void) {
    // Hitung step berdasarkan frekuensi nada dan sampling rate target
    uint32_t step = (uint32_t)(((uint64_t)g_tone_freq_hz << 32) / g_sample_rate_hz);
    
    // Ambil nilai dari tabel sinus atau hasilkan gelombang kotak
    int16_t sample = g_square
                   ? (g_phase & 0x80000000U ? AMPLITUDE : -AMPLITUDE)
                   : (int16_t)(((int32_t)SINE[g_phase >> 27] * AMPLITUDE) / 32767);

    // Kirim ke hardware melalui register data
    write_channels(pack_sample(sample));
    g_phase += step; // Update fase untuk sampel berikutnya
}
\end{lstlisting}

Kombinasi antara sintesis clock pada hardware dan pembangkitan sinyal pada software membentuk sistem audio SoC yang utuh, di mana hardware menyediakan jalur transmisi yang presisi dan software menyediakan konten audio yang fleksibel.

\section{Dampak terhadap Arsitektur Sistem Global}

Implementasi varian DDS memberikan dampak signifikan terhadap penyederhanaan arsitektur sistem pada level *Block Design* Vivado. Meskipun gambar integrasi sistem pada bagian awal bab ini (Gambar \ref{fig:ch4-bd-vivado}) masih menampilkan penggunaan \textit{Clocking Wizard} dengan tiga keluaran clock (sistem, 48 kHz family, dan 44,1 kHz family), penggunaan IP I2S berbasis DDS memungkinkan penghapusan dua jalur clock audio tersebut.

Pada sistem yang teroptimasi, \textit{Clocking Wizard} hanya perlu membangkitkan satu clock sistem (100 MHz). Seluruh sinkronisasi frekuensi audio dilakukan secara internal di dalam IP I2S menggunakan logika DDS. Hal ini tidak hanya mengurangi penggunaan sumber daya *clocking* pada FPGA, tetapi juga menyederhanakan proses *routing* dan meminimalisir potensi masalah *timing closure* pada domain clock yang berbeda. Ringkasan perbedaan arsitektur ini dijelaskan lebih lanjut pada bagian perbandingan di bawah ini.

\section{Ringkasan Implementasi}

Seluruh tahapan perancangan dan implementasi yang telah dijelaskan membentuk sebuah sistem peripheral I2S transmitter yang komplit. Dimulai dari perancangan arsitektur berbasis *fixed-clock* sebagai desain dasar, hingga pengembangan varian DDS yang lebih fleksibel dan mandiri. Meskipun terdapat perbedaan pada mekanisme pembangkitan clock dan sinkronisasi perangkat lunak, kedua desain tetap mematuhi standar protokol Philips I2S yang sama. Dengan demikian, hasil pengujian sinyal dan audio yang disajikan pada Bab V tetap relevan untuk memvalidasi fungsionalitas inti dari IP I2S yang dirancang dalam penelitian ini.
