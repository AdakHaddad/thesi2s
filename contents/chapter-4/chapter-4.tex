    [get_ports i2s_ws_0]

    [get_ports i2s_data_0]

## Configuration options for Basys3
set_property CONFIG_VOLTAGE 3.3 [current_design]
set_property CFGBVS VCCO [current_design]
set_property BITSTREAM.GENERAL.COMPRESS TRUE [current_design]
set_property BITSTREAM.CONFIG.CONFIGRATE 33 [current_design]

\subsection{Detail logika RTL (serializer dan CDC)}

Logika internal IP I2S TX dirancang untuk menangani dua domain clock yang berbeda secara aman. Bagian ini merinci mekanisme tersebut.

\subsubsection{Mekanisme Clock Domain Crossing (CDC)}

Data audio ditulis oleh MicroBlaze V pada domain \texttt{S\_AXI\_ACLK} (100 MHz), sementara serializer beroperasi pada domain \texttt{internal\_mclk} (${\sim}24$ MHz). Untuk mencegah metastabilitas, register \texttt{CONTROL}, \texttt{DATA\_LEFT}, dan \texttt{DATA\_RIGHT} dilewatkan melalui modul \texttt{cdc\_sync} yang mengimplementasikan sinkronisasi flip-flop ganda (\textit{two-stage synchronizer})~\citep{ginosar_metastability_2011}. Selain itu, latching sampel ke register bayangan (\textit{shadow register}) dilakukan tepat pada batas frame stereo (saat \texttt{bit\_count} kembali ke nol) untuk menjamin integritas data antar kanal.

\subsubsection{Logika serializer Philips I2S}

Serializer diimplementasikan sebagai mesin state sederhana yang dipicu oleh \texttt{bclk\_en\_w}. Fungsi \texttt{i2s\_next\_serial\_bit} menentukan bit mana yang harus dikirim berdasarkan nilai \texttt{bit\_count\_q} (0--63):
\begin{itemize}
    \item \textbf{Bit 0 dan 32}: Merupakan bit tunda (\textit{one-bit delay}) sesuai standar Philips; output dipaksa nol.
    \item \textbf{Bit 1--31 dan 33--63}: Mengirimkan bit sampel dari MSB ke LSB. Jika \texttt{SAMPLE\_WIDTH} kurang dari 32, sisa bit dalam slot akan diisi nol (\textit{zero padding}).
\end{itemize}

\noindent Sinyal \texttt{WS} (Word Select) diturunkan langsung dari bit ke-6 dari \texttt{bit\_count\_q}, sehingga otomatis bernilai rendah untuk 32 bit pertama (kiri) dan tinggi untuk 32 bit berikutnya (kanan). Data diperbarui pada tepi turun BCLK dan dipertahankan stabil saat tepi naik BCLK untuk menjamin waktu \textit{setup} dan \textit{hold} pada sisi DAC.

\begin{itemize}
    \item Satu frame stereo terdiri dari tepat 64 periode BCLK (32 kiri + 32 kanan).
    \item WS beralih satu BCLK \textit{sebelum} MSB data, sesuai spesifikasi Philips.
    \item Bit pertama setelah setiap peralihan WS selalu bernilai 0 (\textit{delay bit} wajib).
    \item BCLK = 3,0746 MHz dan WS = 48,03 kHz (konsisten dengan clock aktual 24,597 MHz dan divider 8).
\end{itemize}

\section{Identifikasi dan Perbaikan Bug Firmware}
% ============================================================

Selama validasi perangkat keras ditemukan lima gejala tidak benar yang semuanya bersumber dari \textbf{kode firmware} (\texttt{helloworld.c}), bukan dari RTL. Berikut adalah analisis dan solusi untuk masing-masing bug.

% ------------------------------------------------------------
\subsection{Bug 1: Masker Merusak Bit Tanda Sampel}
% ------------------------------------------------------------

\subsubsection{Gejala}
Amplitudo gelombang kanal kiri dan kanan tidak sama meskipun nilai yang dikirim identik. Waveform tampak seperti \textit{half-wave rectification} pada salah satu kanal.

\subsubsection{Penyebab}

RTL mengharapkan MSB sampel berada di bit \texttt{[sample\_width-1]} dari word 32-bit. Kode lama menerapkan masker setelah geseran:

\begin{verbatim}
uint32_t mask = (1U << g_width) - 1U;   // untuk 24-bit: 0x00FFFFFF
uint32_t samp = (uint32_t)s32 & mask;   // memotong bit [31:24]
\end{verbatim}

\noindent Sampel negatif int16 setelah digeser kiri 8 bit memiliki bit tanda di posisi \texttt{[31:24]}. Masker memotong bit-bit ini sehingga nilai negatif menjadi positif -- gelombang sinus terdistorsi.

\subsubsection{Solusi}

Masker dihapus. Geseran disesuaikan agar MSB int16 tepat di bit \texttt{[g\_width-1]}:

\begin{verbatim}
int32_t s32;
if      (g_width >= 32U)  s32 = (int32_t)s << 16;
else if (g_width >  16U)  s32 = (int32_t)s << (g_width - 16U);
else if (g_width <  16U)  s32 = (int32_t)s >> (16U - g_width);
else                       s32 = (int32_t)s;
uint32_t samp = (uint32_t)s32;   // tanpa mask
\end{verbatim}

\noindent RTL hanya membaca bit \texttt{[sample\_width-1:0]} dari register data; bit di atasnya diabaikan, sehingga tidak diperlukan masker pada sisi firmware.

% ------------------------------------------------------------
\subsection{Bug 2: Mode 32-bit Kehilangan 1 Bit Resolusi}
% ------------------------------------------------------------

\subsubsection{Gejala}
Saat \texttt{SAMPLE\_WIDTH=32}, amplitudo gelombang separuh dari yang diharapkan; sampel full-scale positif ($32767$) tidak menghasilkan output maksimum.

\subsubsection{Penyebab}

Untuk \texttt{g\_width=32}, geseran kiri 16 menghasilkan \texttt{0x7FFF0000} untuk sampel \texttt{0x7FFF}. Bit 31 dari word ini adalah 0 (bukan 1). Serializer RTL pada posisi pertama membaca \texttt{sample[31]} sebagai MSB, sehingga seluruh nilai tergeser satu bit ke bawah -- setara dengan kehilangan satu bit resolusi.

\subsubsection{Solusi}

Sama dengan Bug 1 -- perbaikan geseran tanpa masker sudah menyelesaikan masalah ini secara otomatis. \texttt{(int32\_t)s << 16} menghasilkan penempatan MSB yang benar di bit 31.

% ------------------------------------------------------------
\subsection{Bug 3: Kanal Kanan Mute saat Mode Right-Only}
% ------------------------------------------------------------

\subsubsection{Gejala}
Menekan \texttt{R} (Right only): kedua kanal menjadi mute atau tidak terdengar sama sekali.

\subsubsection{Penyebab}

Gejala ini merupakan konsekuensi dari Bug 1. Pada mode \texttt{g\_left=0, g\_right=1}, firmware menulis nilai nol ke \texttt{DATA\_LEFT} dan \texttt{samp} ke \texttt{DATA\_RIGHT}. Karena \texttt{samp} sudah rusak akibat masker (nilai negatif menjadi positif besar, nilai positif tetap), output audio tidak mencerminkan gelombang sinus yang benar. Pada frekuensi tertentu, DC offset yang dihasilkan dapat diinterpretasikan sebagai tidak ada suara oleh DAC.

\subsubsection{Solusi}

Perbaikan Bug 1 secara langsung menyelesaikan gejala ini. Firmware sudah benar dalam menulis \textbf{kedua} register setiap iterasi (baik yang aktif maupun yang bernilai nol), sehingga CDC counter RTL selalu bertambah secara konsisten.

% ------------------------------------------------------------
\subsection{Bug 4: Frekuensi Audio Lebih Rendah dari Target}
% ------------------------------------------------------------

\subsubsection{Gejala}
Frekuensi terukur pada osiloskop atau spektrum audio lebih rendah dari 1 kHz, 2 kHz, dan 5 kHz yang dipilih. Selisih semakin besar pada frekuensi lebih tinggi.

\subsubsection{Penyebab}

Loop utama menggunakan \texttt{usleep(periode - 2)} untuk mengatur laju sampel:

\begin{verbatim}
uint32_t sample_period_us = (1000000U + (sample_rate / 2U)) / sample_rate;
if (sample_period_us > 2) usleep(sample_period_us - 2);
\end{verbatim}

\noindent Terdapat tiga masalah fundamental:
\begin{enumerate}
    \item \textbf{Koreksi tetap \texttt{-2} tidak akurat.} Overhead loop (penulisan AXI, polling UART, cache miss) bervariasi dan tidak selalu tepat 2 µs.
    \item \textbf{Granularitas \texttt{usleep()} terbatas.} Pada lingkungan bare-metal MicroBlaze, resolusi minimum \texttt{usleep()} mendekati 1 µs. Pada 96 kHz (periode 10,4 µs), koreksi 2 µs berarti error frekuensi $\approx 20\%$.
    \item \textbf{Tidak ada akumulasi waktu.} Setiap iterasi mengukur waktu dari nol baru; error per iterasi tidak saling mengkompensasi, melainkan terakumulasi menjadi error frekuensi sistematis.
\end{enumerate}

\subsubsection{Solusi}

Loop diubah menjadi \textbf{deadline-based busy-wait} menggunakan \textit{hardware cycle counter} RISC-V (\texttt{mcycle} CSR)~\citep{riscv_isa_vol1_2019}:

\begin{verbatim}
static inline uint64_t rdcycle64(void)
{
    uint32_t lo, hi, hi2;
    do {
        __asm__ volatile ("rdcycleh %0" : "=r"(hi));
        __asm__ volatile ("rdcycle  %0" : "=r"(lo));
        __asm__ volatile ("rdcycleh %0" : "=r"(hi2));
    } while (hi != hi2);
    return ((uint64_t)hi << 32) | lo;
}
\end{verbatim}

\noindent Counter ini berjalan pada frekuensi CPU (100 MHz), memberikan resolusi 10 ns. Loop utama menggunakan deadline absolut:

\begin{verbatim}
uint64_t t_deadline = rdcycle64();
while (1) {
    uint64_t period_cycles = (uint64_t)XPAR_CPU_CORE_CLOCK_FREQ_HZ 
                             / current_sample_rate();
    stream_sample();
    if (!XUartLite_IsReceiveEmpty(STDIN_BASEADDRESS))
        handle_key((uint8_t)XUartLite_RecvByte(STDIN_BASEADDRESS));

    t_deadline += period_cycles;           // maju satu periode
    while (rdcycle64() < t_deadline) {}   // busy-wait
}\end{verbatim}

\noindent Dengan cara ini, terlepas dari berapa lama operasi dalam iterasi berlangsung, deadline berikutnya selalu berjarak tepat satu periode sampel. Akurasi frekuensi audio menjadi setara dengan akurasi osilator 100 MHz pada board Basys3.

\begin{table}[htbp]
    \centering
    \caption{Perbandingan akurasi timing loop sebelum dan sesudah perbaikan.}
    \label{tab:ch4-timing-compare}
    \begin{tabular}{|l|c|c|}
        \hline
        \textbf{Properti} & \textbf{usleep() lama} & \textbf{rdcycle64() baru} \\
        \hline
        Resolusi waktu & $\approx$1 µs & 10 ns \\
        Kompensasi overhead loop & Tebakan tetap (-2) & Otomatis (deadline absolut) \\

\section{Pembahasan terhadap Tujuan Penelitian}
% ============================================================

\begin{enumerate}
    \item \textbf{Tujuan perancangan IP} tercapai: register AXI4-Lite aktif, data kiri-kanan dimuat dari perangkat lunak, serializer menghasilkan format Philips I2S dengan 64 BCLK per frame stereo, dan BUFGMUX menyeleksi clock keluarga 48/44,1 kHz secara benar.

    \item \textbf{Tujuan integrasi SoC dan Vitis} tercapai: aplikasi bare-metal mengendalikan IP melalui \textit{memory-mapped I/O} (\texttt{Xil\_Out32}/\texttt{Xil\_In32}) untuk inisialisasi, pengisian sampel, dan kontrol mode.

    \item \textbf{Tujuan verifikasi frekuensi dan audio} tercapai setelah perbaikan tiga bug firmware:
    \begin{itemize}
        \item Bug justifikasi bit (masker merusak tanda) diselesaikan dengan menggeser sampel tanpa masker.
        \item Bug mode 32-bit (MSB salah posisi) diselesaikan dengan formula geseran yang diperbaiki.
        \item Bug timing loop (\texttt{usleep} tidak akumulatif) diselesaikan dengan \textit{deadline-based busy-wait} berbasis \texttt{rdcycle64()}.
    \end{itemize}

    \item \textbf{Penting dicatat} bahwa seluruh perbaikan bersifat \textit{firmware-only} --- RTL tidak dimodifikasi. Hal ini mengkonfirmasi bahwa desain RTL sudah benar sejak awal, termasuk mekanisme CDC yang robust dan serializer Philips I2S yang tepat.
\end{enumerate}
