\section{Implementasi RTL dan Integrasi Vivado}
% ============================================================

RTL direalisasikan dalam dua berkas Verilog: \texttt{i2s.v} sebagai \textit{top-level wrapper} dan \texttt{i2s\_slave\_lite\_v1\_0\_S00\_AXI.v} sebagai modul utama yang mencakup seluruh logika AXI4-Lite, CDC, dan serializer I2S sesuai spesifikasi Philips~\citep{philips_i2s_1986}. Integrasi dilakukan melalui Vivado IP Integrator (\textit{Block Design})~\citep{xilinx_vivado_design_suite_2021}; pin I2S dipetakan pada file \texttt{.xdc}.

\noindent Pemisahan antara \texttt{i2s.v} dan \texttt{i2s\_slave\_lite\_v1\_0\_S00\_AXI.v}
dilakukan untuk menjaga tanggung jawab modul tetap jelas. Berkas \texttt{i2s.v} berfungsi
sebagai pembungkus antarmuka tingkat atas dan penghubung ke sistem Vivado, sedangkan berkas
utama menangani detail perilaku peripheral seperti transaksi AXI, sinkronisasi antar-domain
clock, pemilihan clock audio, dan pembangkitan data serial I2S. Pemisahan ini memudahkan proses
debugging karena permasalahan integrasi sistem dapat dibedakan dari permasalahan logika internal
peripheral.

set_property BITSTREAM.GENERAL.COMPRESS TRUE [current_design]
set_property BITSTREAM.CONFIG.CONFIGRATE 33 [current_design]

\end{verbatim}

\noindent Selain pemetaan pin, file constraint juga mendefinisikan hubungan asinkron antara domain clock AXI (100 MHz) dan domain clock audio (~24 MHz) untuk memastikan analisis timing yang benar:

\begin{verbatim}

    -group [get_clocks -quiet internal_mclk_from_44]

set_false_path -from [get_cells -hierarchical *slv_reg2_reg[2]] \
               -to [get_cells -hierarchical *mclk_mux_inst/q*_d1_reg*]
\end{verbatim}

\noindent Ketiga sinyal utama I2S (BCLK, WS/LRCK, DATA) dan MCLK terhubung langsung ke PMOD JA pin 1--4, yang kemudian disambungkan ke modul PCM5102A menggunakan kabel jumper.

tegangan LVCMOS33, dan hubungan antar-domain clock pada file constraint merupakan bagian
integral dari keberhasilan implementasi, bukan sekadar konfigurasi administratif.

\subsection{Detail logika RTL (serializer dan CDC)}

Logika internal IP I2S TX dirancang untuk menangani dua domain clock yang berbeda secara aman. Bagian ini merinci mekanisme tersebut.

Sebelum data mencapai serializer, register sampel dan kontrol terlebih dahulu
diisi melalui jalur AXI4-Lite. Cuplikan logika tulis register ditunjukkan pada
Listing~\ref{lst:ch4-axi-write}. Listing ini penting karena menunjukkan bahwa
alamat word-aligned \texttt{0x00}, \texttt{0x04}, dan \texttt{0x08} benar-benar
dipetakan ke \texttt{slv\_reg0}, \texttt{slv\_reg1}, dan \texttt{slv\_reg2}.

\begin{lstlisting}[language=Verilog, caption={Cuplikan logika write register AXI4-Lite.}, label={lst:ch4-axi-write}]
always @(posedge S_AXI_ACLK) begin
    if (S_AXI_ARESETN == 1'b0) begin
        slv_reg0 <= 0;
        slv_reg1 <= 0;
        slv_reg2 <= 0;
        slv_reg3 <= 0;
    end else begin
        if (S_AXI_WVALID) begin
            case ((S_AXI_AWVALID) ?
                    S_AXI_AWADDR[ADDR_LSB+OPT_MEM_ADDR_BITS:ADDR_LSB] :
                    axi_awaddr[ADDR_LSB+OPT_MEM_ADDR_BITS:ADDR_LSB])
                REG_DATA_LEFT:
                    ...
                REG_DATA_RIGHT:
                    ...
                REG_CONTROL:
                    ...
                REG_RESERVED:
                    ...
            endcase
        end
    end
end
\end{lstlisting}

\noindent Meskipun sebagian rincian byte-enable disingkat dengan elipsis pada
listing, struktur utamanya tetap terlihat jelas. Setiap transaksi tulis yang
valid akan memilih salah satu register berdasarkan alamat AXI, lalu hanya byte
yang ditandai oleh \texttt{WSTRB} yang diperbarui. Dengan cara ini, peripheral
bersifat kompatibel dengan semantik standar AXI4-Lite dan dapat diakses dari
perangkat lunak menggunakan operasi \texttt{Xil\_Out32()} biasa tanpa memerlukan
protokol tambahan di luar register map.

\subsubsection{Mekanisme Clock Domain Crossing (CDC)}

Data audio ditulis oleh MicroBlaze V pada domain \texttt{S\_AXI\_ACLK} (100 MHz), sementara serializer beroperasi pada domain \texttt{internal\_mclk} (${\sim}24$ MHz). Untuk mencegah metastabilitas, register \texttt{CONTROL}, \texttt{DATA\_LEFT}, dan \texttt{DATA\_RIGHT} dilewatkan melalui modul \texttt{cdc\_sync} yang mengimplementasikan sinkronisasi flip-flop ganda (\textit{two-stage synchronizer})~\citep{ginosar_metastability_2011}. Selain itu, latching sampel ke register bayangan (\textit{shadow register}) dilakukan tepat pada batas frame stereo (saat \texttt{bit\_count} kembali ke nol) untuk menjamin integritas data antar kanal.

Implementasi sinkronisasi tersebut ditunjukkan secara ringkas pada
Listing~\ref{lst:ch4-cdc-sync}. Cuplikan ini memperlihatkan bahwa bus multi-bit
tidak digunakan secara langsung pada domain tujuan, tetapi dilewatkan melalui dua
tahap register berurutan.

\begin{lstlisting}[language=Verilog, caption={Cuplikan modul \texttt{cdc\_sync} pada RTL.}, label={lst:ch4-cdc-sync}]
module cdc_sync #(
    parameter integer WIDTH = 32
) (
    input  wire             dest_clk,
    input  wire [WIDTH-1:0] src_in,
    output wire [WIDTH-1:0] dest_out
);
    (* ASYNC_REG = "TRUE" *) reg [WIDTH-1:0] stage1_q;
    (* ASYNC_REG = "TRUE" *) reg [WIDTH-1:0] stage2_q;

    always @(posedge dest_clk) begin
        stage1_q <= src_in;
        stage2_q <= stage1_q;
    end

    assign dest_out = stage2_q;
endmodule
\end{lstlisting}

\noindent Secara praktis, dua tahap register tersebut memberi waktu agar sinyal
yang mungkin metastabil pada tahap pertama menjadi stabil sebelum digunakan pada
logika domain audio. Pada desain ini, sinkronisasi semacam itu cukup memadai karena
yang dipindahkan antar-domain bukan \textit{stream} kontinu berkecepatan tinggi,
melainkan status kontrol dan data register yang diperbarui secara diskrit oleh CPU.

\subsubsection{Logika serializer Philips I2S}

Serializer diimplementasikan sebagai mesin state sederhana yang dipicu oleh \texttt{bclk\_en\_w}. Fungsi \texttt{i2s\_next\_serial\_bit} menentukan bit mana yang harus dikirim berdasarkan nilai \texttt{bit\_count\_q} (0--63):
\begin{itemize}
    \item \textbf{Bit 0 dan 32}: Merupakan bit tunda (\textit{one-bit delay}) sesuai standar Philips; output dipaksa nol.
    \item \textbf{Bit 1--31 dan 33--63}: Mengirimkan bit sampel dari MSB ke LSB. Jika \texttt{SAMPLE\_WIDTH} kurang dari 32, sisa bit dalam slot akan diisi nol (\textit{zero padding}).
\end{itemize}

\noindent Sinyal \texttt{WS} (Word Select) diturunkan langsung dari bit ke-6 dari \texttt{bit\_count\_q}, sehingga otomatis bernilai rendah untuk 32 bit pertama (kiri) dan tinggi untuk 32 bit berikutnya (kanan). Data diperbarui pada tepi turun BCLK dan dipertahankan stabil saat tepi naik BCLK untuk menjamin waktu \textit{setup} dan \textit{hold} pada sisi DAC.
\noindent Cuplikan logika pemilihan bit serial ditunjukkan pada
Listing~\ref{lst:ch4-serializer}. Struktur ini menjelaskan secara eksplisit
bagaimana aturan \textit{one-bit delay}, pengiriman MSB lebih dulu, dan
\textit{zero padding} direalisasikan pada RTL.

\begin{lstlisting}[language=Verilog, caption={Cuplikan logika serializer I2S.}, label={lst:ch4-serializer}]
case (bit_count_q)
    6'd0, 6'd32: begin
        i2s_next_serial_bit = 1'b0;
    end
    default: begin
        if (bit_count_q < 6'd32) begin
            if ((bit_count_q - 1) < sample_width_q)
                i2s_next_serial_bit = sample_left_q[sample_width_q - bit_count_q];
            else
                i2s_next_serial_bit = 1'b0;
        end else begin
            if ((bit_count_q - 6'd33) < sample_width_q)
                i2s_next_serial_bit = sample_right_q[sample_width_q - (bit_count_q - 6'd32)];
            else
                i2s_next_serial_bit = 1'b0;
        end
    end
endcase
\end{lstlisting}

\noindent Pada bit ke-0 dan ke-32, keluaran dipaksa nol untuk memenuhi
\textit{delay bit} setelah transisi \texttt{WS}. Setelah itu, indeks bit sampel
dihitung mundur dari MSB menuju LSB. Jika \texttt{sample\_width\_q} lebih kecil
dari panjang slot 32-bit, logika secara otomatis mengisi sisa slot dengan nol.
Dengan demikian, satu blok serializer yang sama dapat melayani mode 16, 24, maupun
32 bit tanpa mengubah struktur frame stereo.
\noindent Dengan mekanisme tersebut, perilaku serializer dapat dipandang sebagai rantai proses
yang deterministik: sampel baru dari domain AXI tidak langsung dikirim ke jalur serial, tetapi
ditahan hingga frame yang sedang berjalan selesai. Strategi ini mencegah terjadinya pergantian
nilai di tengah slot yang dapat menyebabkan sebagian bit kanal kiri berasal dari sampel lama dan
sebagian bit berikutnya berasal dari sampel baru. Selain itu, kombinasi \texttt{ENABLE},
\texttt{MUTE}, dan \texttt{SAMPLE\_WIDTH} memungkinkan satu arsitektur serializer yang sama
dipakai untuk beberapa mode operasi tanpa perlu mengubah struktur dasar frame 64-bit.

\section{Pengujian Sinyal}
% ============================================================

Pengamatan sinyal pada \texttt{i2s\_bclk}, \texttt{i2s\_ws}, dan \texttt{i2s\_data} dilakukan menggunakan alat ukur digital (osiloskop/logic analyzer) untuk memverifikasi timing Philips I2S~\citep{philips_i2s_1986}.

\begin{figure}[htbp]
    \centering
    \IfFileExists{contents/figures/sample-latching-leftright-waveform.png}{%
        \includegraphics[width=0.95\textwidth]{contents/figures/sample-latching-leftright-waveform.png}%
    }{}
    \caption{Tangkapan waveform: BCLK, WS, dan DATA -- satu frame stereo 64 BCLK.}
    \label{fig:ch4-ila}
\end{figure}

\noindent Verifikasi ILA mengkonfirmasi:
\begin{itemize}
    \item Satu frame stereo terdiri dari tepat 64 periode BCLK (32 kiri + 32 kanan).
    \item WS beralih satu BCLK \textit{sebelum} MSB data, sesuai spesifikasi Philips.
    \item Bit pertama setelah setiap peralihan WS selalu bernilai 0 (\textit{delay bit} wajib).
    \item BCLK = 3,0746 MHz dan WS = 48,03 kHz (konsisten dengan clock aktual 24,597 MHz dan divider 8).
\end{itemize}

\noindent Temuan ini penting karena setiap butir di atas memverifikasi aspek yang berbeda dari
desain. Jumlah 64 periode BCLK membuktikan struktur slot 32-bit per kanal, transisi \texttt{WS}
memastikan kompatibilitas terhadap format Philips I2S, sedangkan keberadaan \textit{delay bit}
menunjukkan bahwa serializer tidak sekadar menggeser data mentah, tetapi benar-benar mengikuti
aturan protokol. Kesesuaian antara hasil pengukuran dan perhitungan teoritis juga memperkuat
kesimpulan bahwa jalur clock, divider, dan logika frame bekerja secara konsisten.

\subsection{Verifikasi tambahan menggunakan testbench}

Setelah perilaku dasar IP terverifikasi pada perangkat keras nyata, dilakukan verifikasi

Dengan demikian, testbench berperan sebagai alat dokumentasi dan konfirmasi

\section{Analisis Frekuensi Sampling}
% ============================================================

Bagian ini menganalisis bagaimana frekuensi sampling terbentuk dari clock audio yang
dihasilkan \textit{Clocking Wizard}. Alurnya dapat dibaca secara berurutan sebagai
\texttt{audio\_clk} $\rightarrow$ pembagi internal $\rightarrow$ BCLK $\rightarrow F_s$.
Dengan cara pandang ini, pembaca dapat melihat bahwa frekuensi sampling bukan nilai
yang muncul secara terpisah, melainkan konsekuensi langsung dari clock sumber dan
konfigurasi divider pada IP.

Pada implementasi ini, BCLK dibangkitkan dari \texttt{audio\_clk} melalui pembagi
internal. Hubungannya dinyatakan sebagai berikut:

\begin{equation}
\mathrm{BCLK} = \frac{\mathrm{MCLK}}{\mathrm{Divider}}
\end{equation}

\noindent Setelah BCLK diperoleh, frekuensi sampling dihitung dari jumlah bit dalam
satu frame stereo. Karena format yang digunakan adalah 32 bit per kanal untuk dua
kanal, maka satu frame berisi 64 siklus BCLK. Oleh sebab itu:

\begin{equation}
F_s = \frac{\mathrm{BCLK}}{64}
\end{equation}

\noindent Untuk keluarga frekuensi 48 kHz, clock audio aktual yang digunakan pada
pengujian adalah sekitar 24,597 MHz. Berdasarkan relasi tersebut, hasil perhitungan
untuk masing-masing mode ditunjukkan pada Tabel~\ref{tab:ch4-ukur}.

\begin{table}[H]
    \centering
    \caption{Hasil pengujian clock I2S pada keluarga frekuensi 48 kHz.}
    \label{tab:ch4-ukur}
    \renewcommand{\arraystretch}{1.3}
    \begin{tabular}{|p{2.3cm}|p{2.8cm}|c|c|c|}
        \hline
        \centering\textbf{Mode} &
        \centering\textbf{MCLK} &
        \centering\textbf{Divider} &
        \centering\textbf{BCLK} &
        \centering\textbf{$F_s$} \tabularnewline
        \hline
        \texttt{FS\_MODE = 0} & 24,597 MHz & $\div 8$ & 3,0746 MHz & 48,03 kHz \\
        \hline
        \texttt{FS\_MODE = 1} & 24,597 MHz & $\div 4$ & 6,1493 MHz & 96,08 kHz \\
        \hline
    \end{tabular}
\end{table}

\noindent Tabel~\ref{tab:ch4-ukur} memperlihatkan hubungan yang intuitif. Ketika
\texttt{FS\_MODE = 0}, divider bernilai 8 sehingga BCLK turun menjadi sekitar
3,0746 MHz. Karena satu sampel stereo membutuhkan 64 siklus BCLK, frekuensi sampling
yang dihasilkan menjadi sekitar 48,03 kHz. Ketika \texttt{FS\_MODE = 1}, divider
diperkecil menjadi 4, sehingga BCLK naik hampir dua kali lipat menjadi 6,1493 MHz.
Akibatnya, $F_s$ juga meningkat hampir dua kali lipat menjadi 96,08 kHz.

\noindent Dari hasil tersebut dapat dilihat bahwa perubahan \texttt{FS\_MODE} bekerja
sesuai tujuan rancangan: mode pertama menghasilkan keluarga 48 kHz, sedangkan mode
kedua menghasilkan keluarga 96 kHz. Selisih kecil terhadap nilai nominal berasal dari
frekuensi clock aktual MMCM yang sedikit berbeda dari target ideal 24,576 MHz.
Dengan kata lain, penyimpangan ini bersifat sistematis dan dapat diprediksi dari clock
sumber, bukan disebabkan oleh kesalahan logika serializer atau bug pada RTL. Untuk
tujuan validasi peripheral, hasil ini menunjukkan bahwa jalur clock dan mekanisme
pembagi bekerja secara konsisten serta menghasilkan frekuensi sampling yang sangat
dekat dengan target desain.

Perbaikan dari sisi firmware yang memungkinkan akurasi ini dipertahankan pada saat
streaming ditunjukkan pada Listing~\ref{lst:ch4-rdcycle}. Fungsi
\texttt{rdcycle64()} digunakan untuk membentuk basis waktu absolut pada level siklus CPU.

\begin{lstlisting}[language=C, caption={Cuplikan pembacaan \texttt{mcycle} 64-bit dan loop berbasis deadline.}, label={lst:ch4-rdcycle}]
static inline uint64_t rdcycle64(void)
{
    uint32_t lo, hi, hi2;
    do {
        __asm__ volatile ("rdcycleh %0" : "=r"(hi));
        __asm__ volatile ("rdcycle  %0" : "=r"(lo));
        __asm__ volatile ("rdcycleh %0" : "=r"(hi2));
    } while (hi != hi2);
    return ((uint64_t)hi << 32) | lo;
}

uint64_t t_deadline = rdcycle64();
while (1) {
    uint64_t period_cycles =
        (uint64_t)XPAR_CPU_CORE_CLOCK_FREQ_HZ / current_sample_rate();
    stream_sample();
    t_deadline += period_cycles;
    while (rdcycle64() < t_deadline) {}
}
\end{lstlisting}

\noindent Berbeda dari pendekatan \texttt{usleep()}, loop berbasis deadline tidak
menebak overhead tiap iterasi. Waktu target selalu dimajukan satu periode sampel
secara absolut, lalu CPU menunggu hingga counter mencapai nilai tersebut. Karena itu,
error waktu tidak terakumulasi dari satu iterasi ke iterasi berikutnya. Inilah alasan
utama mengapa frekuensi hasil perbaikan menjadi jauh lebih dekat dengan nilai teoritis.

\section{Identifikasi dan Perbaikan Bug Firmware}
% ============================================================

Selama validasi perangkat keras ditemukan empat gejala tidak benar yang semuanya bersumber dari \textbf{kode firmware} (\texttt{helloworld.c}), bukan dari RTL. Berikut adalah analisis dan solusi untuk masing-masing bug.

% ------------------------------------------------------------
\subsection{Bug 1: Masker Merusak Bit Tanda Sampel}
% ------------------------------------------------------------

\subsubsection{Gejala}
Amplitudo gelombang kanal kiri dan kanan tidak sama meskipun nilai yang dikirim identik. Waveform tampak seperti \textit{half-wave rectification} pada salah satu kanal.

\subsubsection{Penyebab}

RTL mengharapkan MSB sampel berada di bit \texttt{[sample\_width-1]} dari word 32-bit. Kode lama menerapkan masker setelah geseran:

\begin{lstlisting}[language=C,basicstyle=\ttfamily\small,breaklines=true,columns=fullflexible]
uint32_t mask = (1U << g_width) - 1U;   // untuk 24-bit: 0x00FFFFFF
uint32_t samp = (uint32_t)s32 & mask;   // memotong bit [31:24]
\end{lstlisting}

\noindent Sampel negatif int16 setelah digeser kiri 8 bit memiliki bit tanda di posisi \texttt{[31:24]}. Masker memotong bit-bit ini sehingga nilai negatif menjadi positif -- gelombang sinus terdistorsi.

\subsubsection{Solusi}

Masker dihapus. Geseran disesuaikan agar MSB int16 tepat di bit \texttt{[g\_width-1]}:

\begin{lstlisting}[language=C,basicstyle=\ttfamily\small,breaklines=true,columns=fullflexible]
int32_t s32;
if      (g_width >= 32U)  s32 = (int32_t)s << 16;
else if (g_width >  16U)  s32 = (int32_t)s << (g_width - 16U);
else if (g_width <  16U)  s32 = (int32_t)s >> (16U - g_width);
else                       s32 = (int32_t)s;
uint32_t samp = (uint32_t)s32;   // tanpa mask
\end{lstlisting}

\noindent RTL hanya membaca bit \texttt{[sample\_width-1:0]} dari register data; bit di atasnya diabaikan, sehingga tidak diperlukan masker pada sisi firmware.

% ------------------------------------------------------------
\subsection{Bug 2: Mode 32-bit Kehilangan 1 Bit Resolusi}
% ------------------------------------------------------------

\subsubsection{Gejala}
Saat \texttt{SAMPLE\_WIDTH=32}, amplitudo gelombang separuh dari yang diharapkan; sampel full-scale positif ($32767$) tidak menghasilkan output maksimum.

\subsubsection{Penyebab}

Untuk \texttt{g\_width=32}, geseran kiri 16 menghasilkan \texttt{0x7FFF0000} untuk sampel \texttt{0x7FFF}. Bit 31 dari word ini adalah 0 (bukan 1). Serializer RTL pada posisi pertama membaca \texttt{sample[31]} sebagai MSB, sehingga seluruh nilai tergeser satu bit ke bawah -- setara dengan kehilangan satu bit resolusi.

\subsubsection{Solusi}

Sama dengan Bug 1 -- perbaikan geseran tanpa masker sudah menyelesaikan masalah ini secara otomatis. \texttt{(int32\_t)s << 16} menghasilkan penempatan MSB yang benar di bit 31.

% ------------------------------------------------------------
\subsection{Bug 3: Kanal Kanan Mute saat Mode Right-Only}
% ------------------------------------------------------------

\subsubsection{Gejala}
Menekan \texttt{R} (Right only): kedua kanal menjadi mute atau tidak terdengar sama sekali.

\subsubsection{Penyebab}

Gejala ini merupakan konsekuensi dari Bug 1. Pada mode
\texttt{g\_left=0, g\_right=1}, firmware menulis nol ke \texttt{DATA\_LEFT} dan
\texttt{samp} ke \texttt{DATA\_RIGHT}. Karena \texttt{samp} sudah rusak akibat
masker, output audio tidak lagi mencerminkan gelombang sinus yang benar. Pada
frekuensi tertentu, \textit{DC offset} yang terbentuk bahkan dapat diinterpretasikan
sebagai tidak ada suara oleh DAC.

\subsubsection{Solusi}

Perbaikan Bug 1 secara langsung menyelesaikan gejala ini. Firmware sudah benar dalam menulis \textbf{kedua} register setiap iterasi (baik yang aktif maupun yang bernilai nol), sehingga CDC counter RTL selalu bertambah secara konsisten.

% ------------------------------------------------------------
\subsection{Bug 4: Frekuensi Audio Lebih Rendah dari Target}
% ------------------------------------------------------------

\subsubsection{Gejala}
Frekuensi terukur pada osiloskop atau spektrum audio lebih rendah dari 1 kHz, 2 kHz, dan 5 kHz yang dipilih. Selisih semakin besar pada frekuensi lebih tinggi.

\subsubsection{Penyebab}

Loop utama menggunakan \texttt{usleep(periode - 2)} untuk mengatur laju sampel:

\begin{lstlisting}[language=C,basicstyle=\ttfamily\small,breaklines=true,columns=fullflexible]
uint32_t sample_period_us = (1000000U + (sample_rate / 2U)) / sample_rate;
if (sample_period_us > 2) usleep(sample_period_us - 2);
\end{lstlisting}

\noindent Terdapat tiga masalah fundamental:
\begin{enumerate}
    \item \textbf{Koreksi tetap \texttt{-2} tidak akurat.} Overhead loop (penulisan AXI, polling UART, cache miss) bervariasi dan tidak selalu tepat 2 µs.
    \item \textbf{Granularitas \texttt{usleep()} terbatas.} Pada lingkungan bare-metal MicroBlaze, resolusi minimum \texttt{usleep()} mendekati 1 µs. Pada 96 kHz (periode 10,4 µs), koreksi 2 µs berarti error frekuensi $\approx 20\%$.
    \item \textbf{Tidak ada akumulasi waktu.} Setiap iterasi mengukur waktu dari nol baru; error per iterasi tidak saling mengkompensasi, melainkan terakumulasi menjadi error frekuensi sistematis.
\end{enumerate}

\subsubsection{Solusi}

Loop diubah menjadi \textbf{deadline-based busy-wait} menggunakan \textit{hardware cycle counter} RISC-V (\texttt{mcycle} CSR)~\citep{riscv_isa_vol1_2019}:

\begin{lstlisting}[language=C,basicstyle=\ttfamily\small,breaklines=true,columns=fullflexible]
static inline uint64_t rdcycle64(void)
{
    uint32_t lo, hi, hi2;
    do {
        __asm__ volatile ("rdcycleh %0" : "=r"(hi));
        __asm__ volatile ("rdcycle  %0" : "=r"(lo));
        __asm__ volatile ("rdcycleh %0" : "=r"(hi2));
    } while (hi != hi2);
    return ((uint64_t)hi << 32) | lo;
}
\end{lstlisting}

\noindent Counter ini berjalan pada frekuensi CPU (100 MHz), memberikan resolusi 10 ns. Loop utama menggunakan deadline absolut:

\begin{lstlisting}[language=C,basicstyle=\ttfamily\small,breaklines=true,columns=fullflexible]
uint64_t t_deadline = rdcycle64();
while (1) {
    uint64_t period_cycles = (uint64_t)XPAR_CPU_CORE_CLOCK_FREQ_HZ 
                             / current_sample_rate();
    stream_sample();
    if (!XUartLite_IsReceiveEmpty(STDIN_BASEADDRESS))
        handle_key((uint8_t)XUartLite_RecvByte(STDIN_BASEADDRESS));

    t_deadline += period_cycles;           // maju satu periode
    while (rdcycle64() < t_deadline) {}   // busy-wait
}
\end{lstlisting}

\noindent Dengan cara ini, terlepas dari berapa lama operasi dalam iterasi berlangsung, deadline berikutnya selalu berjarak tepat satu periode sampel. Akurasi frekuensi audio menjadi setara dengan akurasi osilator 100 MHz pada board Basys3.

\section{Pembahasan terhadap Tujuan Penelitian}
% ============================================================

\begin{enumerate}
    \item \textbf{Tujuan perancangan IP} tercapai: register AXI4-Lite aktif, data kiri-kanan dimuat dari perangkat lunak, serializer menghasilkan format Philips I2S dengan 64 BCLK per frame stereo, dan BUFGMUX menyeleksi clock keluarga 48/44,1 kHz secara benar.

    \item \textbf{Tujuan integrasi SoC dan Vitis} tercapai: aplikasi bare-metal mengendalikan IP melalui \textit{memory-mapped I/O} (\texttt{Xil\_Out32}/\texttt{Xil\_In32}) untuk inisialisasi, pengisian sampel, dan kontrol mode.

    \item \textbf{Tujuan verifikasi frekuensi dan audio} tercapai setelah perbaikan empat bug firmware:
    \begin{itemize}
        \item Bug justifikasi bit (masker merusak tanda) diselesaikan dengan menggeser sampel tanpa masker.
        \item Bug mode 32-bit (MSB salah posisi) diselesaikan dengan formula geseran yang diperbaiki.
        \item Bug kanal kanan pada mode \textit{right-only} terselesaikan setelah representasi sampel kembali benar.
        \item Bug timing loop (\texttt{usleep} tidak akumulatif) diselesaikan dengan \textit{deadline-based busy-wait} berbasis \texttt{rdcycle64()}.
    \end{itemize}

    \item \textbf{Penting dicatat} bahwa seluruh perbaikan bersifat \textit{firmware-only} --- RTL tidak dimodifikasi. Hal ini mengkonfirmasi bahwa desain RTL sudah benar sejak awal, termasuk mekanisme CDC yang robust dan serializer Philips I2S yang tepat.
\end{enumerate}

\noindent Secara keseluruhan, hasil penelitian menunjukkan bahwa pendekatan register-based
AXI4-Lite cukup efektif untuk membangun peripheral I2S TX yang sederhana, mudah diintegrasikan,
dan mudah dianalisis. Keterbatasan utama sistem ini terletak pada tidak adanya FIFO atau DMA,
sehingga kontinuitas sampel masih bergantung pada kemampuan perangkat lunak mempertahankan laju
tulis. Meski demikian, untuk tujuan tugas akhir yang menekankan ketepatan protokol, integrasi
SoC, dan keterlacakan proses debugging, arsitektur yang dihasilkan sudah memadai dan sesuai
dengan sasaran penelitian.
