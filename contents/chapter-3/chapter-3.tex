\section{Spesifikasi dan arsitektur IP I2S TX}

Bagian ini merangkum \textbf{perancangan fungsional} IP yang menjadi inti skripsi. Implementasi difokuskan pada peripheral \textbf{I2S transmitter} dengan antarmuka \textbf{AXI4-Lite} agar dapat dikontrol oleh program C pada MicroBlaze V~\citep{xilinx_microblaze_ref_2021}.

\subsection{Daftar fitur IP}
\begin{itemize}
    \item Antarmuka \textbf{AXI4-Lite} 32 bit untuk konfigurasi dan pengisian data audio~\citep{arm_amba_axi_2010}.
    \item Mode \textbf{TX-only}, stereo, format \textbf{Philips I2S}~\citep{philips_i2s_1986}, \textbf{24 bit} per saluran dalam \textbf{slot 32 bit}.
    \item Dukungan dua keluarga frekuensi sampling nominal (\textbf{48/96 kHz} dan \textbf{44,1/88,2 kHz}), melalui pembagi clock internal dan \texttt{BUFGMUX}~\citep{xilinx_clocking_wizard_2021}.
    \item \textbf{Jalur data} lewat register \texttt{DATA\_LEFT}/\texttt{DATA\_RIGHT} dan \texttt{CONTROL}, \textbf{tanpa FIFO} internal.
    \item Keluaran menuju modul \textbf{PCM5102A}~\citep{pcm5102a_datasheet}: BCLK, LRCK/WS, DATA, dan MCLK bila digunakan.
\end{itemize}

\subsection{Aturan packing data}
\begin{itemize}
\item \texttt{DATA\_LEFT}: sampel kanal kiri ditulis dengan \textbf{MSB di bit \texttt{[SAMPLE\_WIDTH-1]}} dari word 32-bit. Bit di atas \texttt{SAMPLE\_WIDTH-1} diabaikan oleh RTL.
\item \texttt{DATA\_RIGHT}: aturan packing sama dengan \texttt{DATA\_LEFT}, untuk sampel kanal kanan.
\item Firmware harus selalu menulis \textbf{kedua} register setiap periode sampel agar counter CDC RTL bertambah secara konsisten untuk kedua kanal.
\item Contoh konversi untuk sampel int16: geser kiri sebesar \texttt{(SAMPLE\_WIDTH - 16)} bit sehingga MSB tepat di bit \texttt{[SAMPLE\_WIDTH-1]}, \textbf{tanpa masker tambahan}.
\item Jika \texttt{MUTE=1}, serializer memaksa keluaran DATA ke nol tanpa mengubah isi register sampel.
\end{itemize}

\noindent Sebagai contoh, jika firmware menghasilkan sampel bertipe \texttt{int16\_t} bernilai
\texttt{-32768} dan sistem dikonfigurasi dengan \texttt{SAMPLE\_WIDTH=24}, maka sampel tersebut
harus digeser ke kiri 8 bit agar bit tanda berada tepat pada posisi \texttt{[23]}. Dengan
demikian, RTL dapat langsung mengambil bit \texttt{[23:0]} sebagai data valid untuk slot I2S,
sedangkan bit di atasnya diabaikan. Penjelasan ini penting karena kesalahan kecil pada posisi
MSB atau penggunaan masker yang tidak tepat dapat mengubah arti data bertanda, meskipun format
register pada sisi AXI tetap tampak benar.

Strategi pembentukan \texttt{CONTROL} dari sisi perangkat lunak ditunjukkan pada
Listing~\ref{lst:ch3-apply-control}. Cuplikan ini penting karena memperlihatkan
hubungan langsung antara variabel state firmware dan bitfield yang telah
didefinisikan pada register \texttt{CONTROL}.

\begin{lstlisting}[language=C, caption={Cuplikan fungsi \texttt{apply\_control()} pada firmware.}, label={lst:ch3-apply-control}]
#define CTRL_ENABLE   (1U << 0)
#define CTRL_MUTE     (1U << 1)
#define CTRL_FS       (1U << 2)
#define CTRL_FS_MODE  (1U << 3)
#define CTRL_WIDTH(w) (((uint32_t)(w) & 0x1FU) << 8U)

static void apply_control(void)
{
    uint32_t ctrl = 0U;
    if (g_enable)  ctrl |= CTRL_ENABLE;
    if (g_mute)    ctrl |= CTRL_MUTE;
    if (g_fs)      ctrl |= CTRL_FS;
    if (g_fs_mode) ctrl |= CTRL_FS_MODE;
    ctrl |= CTRL_WIDTH(g_width);
    Xil_Out32(CONTROL, ctrl);
}
\end{lstlisting}

\noindent Fungsi tersebut menunjukkan bahwa model kendali peripheral benar-benar
bersifat \textit{register-based}. Firmware tidak perlu memanggil prosedur kompleks
untuk mengganti mode operasi; cukup menyusun word kontrol 32-bit lalu menuliskannya
ke alamat \texttt{CONTROL}. Pendekatan ini membuat perilaku sistem lebih mudah
ditelusuri saat validasi, karena setiap perubahan mode dapat dipetakan langsung ke
bit tertentu pada register.

\section{Alat dan Bahan}

\subsection{Perangkat keras}

\subsubsection{Board FPGA}
\textbf{Basys3} digunakan sebagai platform implementasi utama~\citep{xilinx_basys3_ref}. Board ini berbasis FPGA Xilinx Artix-7 dan menyediakan sumber clock 100 MHz, antarmuka USB-UART, serta header PMOD untuk koneksi ke perangkat eksternal.

\subsubsection{Modul DAC audio}
\textbf{PCM5102A Audio DAC Module} digunakan sebagai penerima data I2S~\citep{pcm5102a_datasheet}. Modul ini mendukung antarmuka audio serial, berbagai \textit{sample rate} hingga 384 kHz, dan keluaran analog stereo untuk verifikasi suara.

\subsubsection{Perangkat bantu}
\begin{itemize}
    \item \textbf{Oscilloscope / logic analyzer / ILA} untuk memverifikasi sinyal BCLK, WS, dan DATA.
    \item \textbf{Speaker atau headphone aktif} untuk uji keluaran audio.
    \item \textbf{Breadboard dan kabel jumper} untuk koneksi Basys3 ke PCM5102A.
\end{itemize}

\subsection{Perangkat lunak}
\begin{itemize}
    \item \textbf{AMD Vivado Design Suite} untuk RTL design, synthesis, implementation, dan integrasi SoC~\citep{xilinx_vivado_design_suite_2021}.
    \item \textbf{Xilinx Vitis} untuk pengembangan perangkat lunak uji dalam bahasa C~\citep{xilinx_vitis_embedded_2021}.
    \item \textbf{Vivado Simulator} atau simulasi lain bila digunakan untuk verifikasi RTL.
    \item \textbf{LaTeX} untuk penulisan dokumen skripsi.
\end{itemize}

\section{Metode yang Digunakan}

\subsection{Metode perancangan hardware}

Perancangan perangkat keras dilakukan dengan tahapan berikut:
\begin{enumerate}
    \item Menentukan spesifikasi I2S TX meliputi format data~\citep{philips_i2s_1986}, lebar sampel, mode frekuensi sampling, dan pemetaan register.
    \item Mendesain arsitektur modul yang mencakup AXI4-Lite slave~\citep{arm_amba_axi_2010}, register data dan kontrol, pembagi clock, serta serializer I2S.
    \item Mengimplementasikan RTL dalam Verilog~\citep{palnitkar_verilog_2003} dengan pemisahan blok sekuensial dan kombinasional secara jelas.
    \item Mengintegrasikan peripheral ke dalam Vivado IP Integrator~\citep{xilinx_vivado_design_suite_2021} bersama MicroBlaze V, BRAM, clocking, dan peripheral pendukung.
\end{enumerate}

\noindent Alur tersebut sejalan dengan pendekatan pengembangan sistem FPGA embedded
yang menempatkan definisi spesifikasi, implementasi RTL, integrasi IP, dan
validasi perangkat lunak sebagai tahapan yang saling terkait, bukan pekerjaan
yang berdiri sendiri~\citep{lafosse_embedded_2018}.

\subsection{Metode verifikasi dan simulasi}

Verifikasi dilakukan pada dua level:
\begin{enumerate}
    \item \textbf{Validasi hardware}, untuk memastikan desain tetap bekerja setelah \textit{synthesis} dan \textit{implementation} pada FPGA nyata.
    \item \textbf{Verifikasi fungsional RTL}, untuk mereproduksi perilaku sinyal I2S, urutan bit, dan logika \texttt{DATA}/\texttt{CONTROL} secara terkontrol melalui simulasi.
\end{enumerate}

Parameter yang diamati meliputi:
\begin{itemize}
    \item relasi timing \texttt{i2s\_bclk}, \texttt{i2s\_ws}, dan \texttt{i2s\_data};
    \item kontinuitas aliran sampel pada skenario uji;
    \item akurasi frekuensi sampling nominal rendah dan tinggi;
    \item keberhasilan keluaran audio pada DAC.
\end{itemize}

\noindent Suatu pengujian dinyatakan lulus apabila empat syarat utama terpenuhi, yaitu:
struktur frame terdiri dari 64 pulsa BCLK per sampel stereo, transisi \texttt{WS} terjadi satu
BCLK sebelum MSB kanal berikutnya, bit tunda setelah transisi \texttt{WS} bernilai nol, dan
DAC mampu menghasilkan audio yang konsisten dengan mode kanal serta frekuensi yang dipilih.
Dengan definisi ini, verifikasi tidak berhenti pada kecocokan waveform semata, tetapi juga
menilai apakah perilaku sistem sudah sesuai dengan tujuan operasional peripheral.

\noindent Pada alur kerja aktual penelitian ini, verifikasi perilaku dasar terlebih dahulu
dilakukan melalui \textit{bitstream bring-up}, pengamatan ILA, dan pengukuran pada perangkat
keras. Setelah hubungan timing utama telah teridentifikasi pada FPGA nyata, dibuat
\textit{testbench} Vivado/XSim pada proyek \texttt{C:\textbackslash tesi2s\textbackslash testblock}
untuk mereproduksi kondisi \texttt{enable}, \texttt{mute}, pemilihan \texttt{fs\_mode}, dan
serialisasi sampel kiri-kanan secara lebih formal. Dengan demikian, simulasi pada penelitian ini
berfungsi sebagai verifikasi pelengkap yang mendokumentasikan ulang perilaku yang telah diamati
pada perangkat keras, bukan sebagai satu-satunya sumber validasi.

Di sisi firmware, pembangkitan sampel uji sekaligus penulisan ke register data
ditunjukkan pada Listing~\ref{lst:ch3-stream-sample}. Cuplikan ini menjadi
penghubung penting antara teori packing data dan cara perangkat lunak benar-benar
memasukkan sampel ke peripheral.

\begin{lstlisting}[language=C, caption={Cuplikan fungsi \texttt{stream\_sample()} untuk packing dan penulisan sampel.}, label={lst:ch3-stream-sample}]
int32_t s32;
if (g_width >= 32U) {
    s32 = (int32_t)s << 16;
} else if (g_width > 16U) {
    s32 = (int32_t)s << (g_width - 16U);
} else if (g_width < 16U) {
    s32 = (int32_t)s >> (16U - g_width);
} else {
    s32 = (int32_t)s;
}
uint32_t samp = (uint32_t)s32;

Xil_Out32(DATA_LEFT,  g_left  ? samp : 0U);
Xil_Out32(DATA_RIGHT, g_right ? samp : 0U);
\end{lstlisting}

\noindent Dua hal dapat diamati dari listing tersebut. Pertama, posisi MSB sampel
selalu disesuaikan terhadap \texttt{g\_width}, sehingga format data pada sisi
firmware konsisten dengan ekspektasi serializer RTL. Kedua, kedua register
\texttt{DATA\_LEFT} dan \texttt{DATA\_RIGHT} selalu ditulis pada setiap iterasi,
termasuk ketika salah satu kanal di-nol-kan. Keputusan ini sesuai dengan rancangan
CDC pada hardware yang melatch pasangan sampel kiri-kanan secara bersamaan pada
batas frame.

\subsection{Metode integrasi SoC}

Integrasi peripheral ke dalam SoC dilakukan melalui alur berikut:
\begin{enumerate}
    \item \textbf{IP Packaging}: mengemas modul I2S sebagai custom IP menggunakan Vivado IP Integrator~\citep{xilinx_vivado_design_suite_2021}.
    \item \textbf{Block Design}: menghubungkan MicroBlaze V~\citep{xilinx_microblaze_ref_2021}, AXI interconnect~\citep{xilinx_axi_ref_2017}, BRAM, Clocking Wizard~\citep{xilinx_clocking_wizard_2021}, UART, dan IP I2S.
    \item \textbf{Address Allocation}: menetapkan base address peripheral agar dapat diakses dari Vitis~\citep{xilinx_vitis_embedded_2021}.
    \item \textbf{Constraint Management}: memetakan pin I2S ke konektor board sesuai file \texttt{.xdc}~\citep{xilinx_basys3_ref}.
\end{enumerate}

\noindent Referensi mengenai \textit{custom IP} penting pada tahap ini karena
integrasi peripheral pada Vivado tidak hanya memerlukan RTL yang berfungsi, tetapi
juga metadata IP, antarmuka bus yang dikenali tool, dan kompatibilitas dengan
 mekanisme packaging pada IP Integrator~\citep{xilinx_vivado_design_suite_2021}. Oleh sebab itu,
keberhasilan integrasi SoC pada penelitian ini bergantung pada dua aspek sekaligus,
yaitu kebenaran logika internal peripheral dan ketepatan pengemasannya sebagai IP
yang dapat dikonsumsi oleh Block Design.

\subsection{Metode pengembangan software}

Perangkat lunak uji dikembangkan dalam bahasa C pada Vitis dengan fungsi utama:
\begin{itemize}
    \item menginisialisasi register peripheral I2S;
    \item menulis sampel audio ke register \texttt{DATA\_LEFT} dan \texttt{DATA\_RIGHT};
    \item mengaktifkan mode operasi dan memilih frekuensi sampling;
    \item menyinkronkan waktu penulisan data ke dalam register secara langsung (\textit{programmed I/O});
    \item menghasilkan sinyal uji sederhana seperti gelombang sinus atau \textit{square wave}.
\end{itemize}

\subsection{Metode pengujian hardware}

Pengujian hardware dilakukan dengan langkah berikut:
\begin{enumerate}
    \item Memprogram FPGA dengan bitstream hasil implementasi.
    \item Menghubungkan Basys3 ke PCM5102A.
    \item Menjalankan aplikasi Vitis untuk mengirim sampel audio.
    \item Mengamati sinyal I2S menggunakan ILA atau osiloskop.
    \item Memverifikasi keluaran audio melalui speaker/headphone.
\end{enumerate}

\noindent Skenario uji perangkat keras disusun agar mencakup fungsi-fungsi utama peripheral,
yakni pengaktifan dan penonaktifan output, pengujian mode \texttt{MUTE}, pemisahan kanal kiri
dan kanan, perubahan \texttt{SAMPLE\_WIDTH}, serta pemilihan keluarga frekuensi sampling.
Setiap skenario diamati pada dua level sekaligus: level sinyal digital menggunakan ILA atau
osiloskop untuk memverifikasi ketepatan protokol, dan level audio analog untuk memastikan bahwa
konfigurasi register benar-benar menghasilkan perilaku yang diharapkan pada DAC.

\subsection{Pengukuran dan evaluasi}

Metode evaluasi meliputi:
\begin{itemize}
    \item \textbf{Resource utilization}: LUT, FF, dan BRAM dari laporan implementasi.
    \item \textbf{Clock accuracy}: frekuensi aktual hasil Clocking Wizard dan frekuensi terukur sinyal I2S.
    \item \textbf{Fungsionalitas audio}: keberhasilan playback dan kontinuitas sampel yang memadai pada skenario uji utama.
\end{itemize}
