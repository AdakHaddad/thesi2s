\chapter{Hasil dan Pembahasan}

Bab ini menyajikan hasil pengujian fungsional sistem IP I2S Transmitter yang dikembangkan, mencakup verifikasi timing protokol, analisis akurasi frekuensi sampling, identifikasi dan perbaikan bug firmware, serta pengujian integrasi dengan blok pemrosesan sinyal digital eksternal. Seluruh pengukuran dilakukan pada perangkat keras target Basys3 (FPGA Artix-7 XC7A35T) menggunakan osiloskop digital untuk sinyal analog dan capture timing digital untuk sinyal internal.

\begin{enumerate}
	\item Satu frame stereo terdiri atas tepat 64 pulsa BCLK (32 slot kiri, 32 slot kanan).
	\item Transisi WS terjadi bersamaan dengan bit pertama setelah batas frame, yaitu bit delay bernilai nol.

	\item Penghitung bit (\texttt{bit\_count\_q}) kembali ke nol tepat di batas frame stereo, memastikan sinkronisasi awal kanal kiri pada setiap frame.
\end{enumerate}

\section{Analisis Frekuensi Sampling}
% ============================================================

Frekuensi sampling dikonfigurasi melalui kombinasi bit \path{FS\_FAMILY} dan \path{FS\_MODE} pada register \path{CONTROL}. Dua sumber clock dari MMCM menyediakan referensi untuk keluarga 48~kHz (24,576~MHz) dan 44,1~kHz (22,579~MHz). Clock yang aktif dipilih oleh logika seleksi clock internal berdasarkan bit \path{FS\_FAMILY}, sedangkan \path{FS\_MODE} menentukan faktor pembagi BCLK sehingga menghasilkan empat nilai $F_s$ standar.

\section{Validasi Audio pada Berbagai Frekuensi Sampling}
% ============================================================

Pengujian fungsional dilakukan dengan memutar nada sinus menggunakan firmware berbasis DDS perangkat lunak (\textit{software DDS}) pada empat mode $F_s$ standar secara bergantian. Peralihan $F_s$ dilakukan secara langsung melalui penulisan bit \path{FS\_FAMILY} dan \path{FS\_MODE} pada register \path{CONTROL} tanpa reset hardware.

DAC PCM5102A berhasil melakukan \textit{lock} pada setiap transisi $F_s$ secara instan tanpa menghasilkan distorsi \textit{pitch} maupun \textit{audio pop} yang terdengar pada headset. Hasil ini mengonfirmasi bahwa mekanisme penghentian stream yang bersih (\textit{clean frame boundary}) saat peralihan \path{ENABLE} bekerja sebagaimana dirancang, sehingga DAC tidak menerima frame yang terpotong di tengah.

\section{Identifikasi dan Perbaikan Bug Firmware}
% ============================================================

Selama tahap integrasi perangkat keras ditemukan dua bug pada firmware (\path{helloworld.c}) yang memengaruhi kualitas audio dan akurasi frekuensi nada. Keduanya berhasil diidentifikasi dan diperbaiki sebelum pengambilan data final.

\subsection{Bug 1: Masker Bit Merusak Bit Tanda Sampel}

\textbf{Gejala.} Gelombang audio terdengar terdistorsi dengan amplitudo asimetris antara setengah siklus positif dan negatif.

\textbf{Penyebab.} Operasi masker bitwise pada sampel \texttt{int16\_t} dilakukan setelah pengeseran kiri ke posisi bit slot 24-bit. Masker berbasis lebar bit (\texttt{(1 << g\_width) - 1}) memotong bit tanda MSB hasil pengeseran, sehingga sampel negatif kehilangan bit tandanya dan diinterpretasikan sebagai nilai besar positif oleh DAC.

\textbf{Solusi.} Masker dihapus. Sampel \texttt{int16\_t} di-\textit{cast} ke \texttt{int32\_t} sebelum pengeseran agar ekstensi tanda terjaga, kemudian dikirim langsung ke register data tanpa pemotongan bit tanda.

\subsection{Bug 2: Timing Loop Berbasis \texttt{usleep()}}

\textbf{Gejala.} Frekuensi nada keluaran tidak stabil (\textit{jitter} tinggi) dan konsisten lebih rendah dari target. Pada mode 48~kHz firmware yang menggunakan \texttt{usleep(20)} seharusnya menghasilkan rate 50.000 sampel/s, namun overhead fungsi \texttt{usleep()} menyebabkan rate aktual lebih rendah sehingga frekuensi nada ikut turun.

\textbf{Penyebab.} Fungsi \texttt{usleep()} pada MicroBlaze bare-metal memiliki resolusi dan akurasi yang bergantung pada implementasi BSP, dan tidak tersinkronisasi dengan transisi frame hardware I2S.

\textbf{Solusi.} Penulisan sampel disinkronkan dengan sinyal latch hardware menggunakan \textit{status-bit polling} pada bit \path{STATUS[0]} (Listing~\ref{lst:ch5-timing-fix}). Firmware menunggu perubahan bit latch sebelum menulis sampel berikutnya, sehingga laju transfer data secara fisik mengikuti clock hardware tanpa overhead fungsi tunda.

\begin{lstlisting}[
	language=C,
	caption={Sinkronisasi penulisan sampel menggunakan status-bit polling.},
	label={lst:ch5-timing-fix}
	]
	while (1) {
		/* Tunggu sinyal latch baru dari hardware (toggle setiap frame) */
		while (Xil_In32(STATUS_REG) == g_last_status);
		g_last_status = Xil_In32(STATUS_REG);
		
		/* Kirim sampel tepat setelah hardware siap menerima */
		stream_software_dds_sample();
	}
\end{lstlisting}

\subsection{Variabel dan Fungsi Kunci Firmware}

Fungsionalitas utama firmware dikendalikan melalui beberapa komponen berikut:

\begin{itemize}
	\item \path{g\_phase} — akumulator fase 32-bit untuk DDS perangkat lunak; bertambah sebesar \textit{phase step} setiap sampel.
	\item \path{g\_sample\_rate\_hz} — menyimpan nilai $F_s$ aktif; digunakan untuk menghitung \textit{phase step} agar akurasi frekuensi nada tidak bergantung pada mode $F_s$ yang dipilih.
	\item \path{frame\_status\_bit()} — membaca bit[0] register \path{RESERVED} yang direfleksikan dari sinyal toggle \texttt{mclk\_latch\_toggle\_q} di domain audio; digunakan sebagai sinyal sinkronisasi.
	\item \path{pack\_sample()} — melakukan penskalaan dan pengeseran sampel sesuai lebar bit yang aktif tanpa memotong bit tanda.
	\item \path{apply\_control()} — mengemas seluruh bit konfigurasi ke dalam satu nilai 32-bit dan menulis ke register \path{CONTROL} dalam satu transaksi AXI4-Lite.
\end{itemize}

\section{Verifikasi Integrasi dengan Filter Digital}
% ============================================================

\input{contents/chapter-5/subsection-i2s-filter}

Pengujian integrasi memakai blok filter IIR (\textit{Infinite Impulse Response}) berbasis RTL yang dikembangkan oleh Adji Pangestu pada penelitian terkait \citep{pangestu_iir_filter}. Pengujian berfokus pada pertukaran data antara aliran serial I2S dan antarmuka paralel filter serta pemeliharaan sinkronisasi frame.

Konfigurasi sistem pada Gambar~\ref{fig:ch5-filter-integration} terdiri atas prosesor MicroBlaze V, AXI SmartConnect, IP I2S hasil penelitian ini, modul \textit{i2s\_filter\_bridge}, serta dua blok \textit{filter\_coeff} untuk kanal kiri dan kanan. IP I2S menghasilkan stream mentah. Bridge mengubah stream tersebut menjadi sampel paralel, menyalurkan setiap kanal ke filter, lalu menyusun keluaran filter menjadi SDATA menuju DAC PCM5102A.

Pengujian menggunakan sinyal sinus 1~kHz sebagai masukan. Capture timing menunjukkan data keluaran filter pada jalur I2S dengan batas kanal yang sama pada setiap frame. Seluruh blok RTL memakai clock sistem 100~MHz, sedangkan bridge memperlakukan BCLK sebagai clock-enable untuk deserialisasi dan serialisasi.

Hasil pengujian menunjukkan bahwa IP I2S dapat menjadi sumber timing dan data bagi rantai pemrosesan sinyal digital eksternal. Konfigurasi $F_s$ melalui register AXI4-Lite tetap menentukan BCLK dan WS karena bridge meneruskan kedua sinyal tersebut tanpa pembangkitan clock tambahan.

\section{Keterbatasan Desain}
% ============================================================

Sebagai bagian dari pembahasan yang komprehensif, berikut adalah keterbatasan yang merupakan ruang lingkup yang telah ditetapkan pada Batasan Penelitian (Bab~I):

\begin{enumerate}
	\item \textbf{TX-only.} IP dirancang sebagai transmitter dan tidak mencakup jalur penerima (RX) maupun mode full-duplex.
	
	\item \textbf{Tanpa DMA.} Transfer data dilakukan melalui \textit{programmed I/O} berbasis register. Streaming audio kontinu memerlukan intervensi CPU setiap frame melalui mekanisme status-bit polling.
	
	\item \textbf{Tanpa FIFO audio.} Tidak terdapat buffer FIFO sehingga potensi \textit{underrun} dikelola sepenuhnya oleh firmware. Pendekatan status-bit polling memitigasi risiko ini namun tidak menghilangkannya sepenuhnya pada kondisi CPU terbebani.
	
	\item \textbf{Tanpa transfer berbasis interupsi.} Firmware menggunakan model polling sinkron sehingga CPU tidak dapat menangani tugas lain selama streaming audio berlangsung.
	
	\item \textbf{Bare-metal only.} Pengujian dilakukan pada lingkungan bare-metal MicroBlaze V tanpa RTOS atau Linux. Integrasi ke ekosistem ASoC Linux merupakan pekerjaan masa mendatang.
\end{enumerate}

Keterbatasan-keterbatasan ini menjadi landasan saran pengembangan yang diuraikan pada Bab~VI, khususnya penambahan FIFO TX, mekanisme DMA berbasis AXI, dan integrasi Linux/ASoC.
